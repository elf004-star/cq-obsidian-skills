# Agent 指令模板

本文档定义了 doc-pipeline 中各类 Agent 的标准指令模板。

---

## ⚠️ 重要：技能联动顺序

**在启动修复 Agent 之前，必须先加载 pandoc-fixer 的 knowledge-base！**

优化后的工作流：

```
1. 读取 pandoc-fixer/references/knowledge-base.md
2. 读取 doc-pipeline/references/fix-rules.md
3. 合并两套规则
4. 启动修复 Agent（携带合并后的完整规则）
5. 完成后直接验收（无需再运行 pandoc-fixer）
```

**原因**：避免修复 Agent 完成后再做 pandoc-fixer 验证时发现新问题，导致二次返工。

---

## 1. 修复 Agent 模板

### 任务描述

```
## 任务：修复文档第 N 部分公式格式

**文件路径**: `{file_path}`（使用相对路径）
**行数范围**: {start_line} - {end_line}
**当前工作目录**: vault 根目录

### 上下文
你是 doc-pipeline 的修复 Agent，负责修复文档中 LaTeX 公式的格式问题。
文档中可能包含数学公式、学术论文、技术文档等内容。

### ⚠️ 必须加载的知识库

在开始修复前，请先读取以下文件获取完整规则：
- `../pandoc-fixer/references/knowledge-base.md`（Pandoc 导出兼容性规则）
- `../doc-pipeline/references/fix-rules.md`（公式格式规范）

### 修复规则

请严格遵守以下规则：

1. **移除字母间空格**
   - `n e t` → `net` 或 `\operatorname{net}`
   - `y ^{i}` → `y^{i}`
   - `w _{i u}` → `w_{iu}`
   - `f _{i}` → `f_{i}`

2. **下标合并**
   - `_{i n _{j}}` → `_{in_j}`
   - `_{c _{j} ^{v}}` → `_{c_j^v}`
   - `^{i n _{j}}` → `^{in_j}`

3. **\frac 格式统一**
   - `\frac {x}{y}` → `\frac{x}{y}`
   - `\frac {\partial}{\partial}` → `\frac{\partial}{\partial}`

4. **\text{} 可读性**
   - `\text{hiddenunit}` → `\text{hidden unit}`
   - `\text{notagate}` → `\text{not a gate}`
   - 单词之间应有空格，便于阅读

5. **括号间距**
   - `\left( expr \right)` → `\left(expr\right)` 或使用 `\,` 添加适当间距
   - `\!\left( x \right)` 用于需要收紧的场合

6. **导数符号**
   - `^{\prime}` 可直接使用
   - `g ^{\prime}` → `g^{\prime}`
   - `h' ` → `h'`（简写形式也可以）

7. **\operatorname 使用**
   - 多字母运算符使用 `\operatorname{op}` 而非分开的字母
   - `n e t` → `\operatorname{net}`

8. **函数名与括号间距**
   - `f (x)` → `f(x)`
   - `\sin (x)` → `\sin(x)`
   - `\exp (- x)` → `\exp(-x)`
   - `g (x)` → `g(x)`
   - 保留必要的数学间距命令：`\!`, `\,` 等

### 禁止事项

- ❌ 不要修改数学含义
- ❌ 不要修改下标变量本身（如 `c_j` 保持为 `c_j`）
- ❌ 不要修改上标内容
- ❌ 不要添加或删除公式中的项
- ❌ 不要使用 Python 脚本处理

### 执行步骤

1. 使用 Read 工具读取文件内容
2. 定位到指定行数范围
3. 逐行检查公式格式
4. 使用 Edit 工具修复问题
5. 生成修改报告

### 输出格式

修复完成后，请输出：

```markdown
## 修复报告 - Part N

**文件**: `{file_path}`
**行数**: {start} - {end}

### 修改统计
- 修改行数: N
- 修改点数: M

### 修改详情
| 行号 | 修复前 | 修复后 |
|------|--------|--------|
| 123 | `n e t` | `\operatorname{net}` |
| 456 | `_{i n}` | `_{in}` |

### 发现但无法修复的问题
（如有）
- 行 789: [描述问题和原因]
```

### 开始执行

请现在开始修复第 {start_line} - {end_line} 行的内容。
```

## 2. 审核 Agent 模板

### 任务描述

```
## 任务：审核修复后的文档

**文件路径**: `{file_path}`（使用相对路径）

### 审核清单

请检查以下所有项目：

#### 1. 字母间距检查
- [ ] 无 `n e t` 等字母间空格
- [ ] 无 `y ^{i}` 中的空格（应为 `y^{i}`）
- [ ] 无 `w _{i u}` 中的空格（应为 `w_{iu}`）

#### 2. 下标配对检查
- [ ] 无 `_{i n _{j}}` 等分散下标（应为 `_{in_j}`）
- [ ] 无 `^{i n _{j}}` 等分散上标（应为 `^{in_j}`）

#### 3. \frac 格式检查
- [ ] 所有 `\frac` 使用统一格式 `\frac{x}{y}`
- [ ] 无 `\frac {` 格式（空格应在括号内）

#### 4. \text{} 可读性检查
- [ ] `\text{}` 内部单词间有空格
- [ ] 无 `\text{hiddenunit}`（应为 `\text{hidden unit}`）

#### 5. 语法正确性检查
- [ ] 括号匹配正确（`\left` 有对应 `\right`）
- [ ] 无未闭合的括号
- [ ] 下标格式正确（`_{xxx}` 而非 `_{ xxx}`）

#### 6. 语义完整性检查
- [ ] 数学表达式未被误改
- [ ] 变量名未被改变
- [ ] 公式逻辑完整

### 问题分类

将发现的问题分类：

| 类型 | 说明 |
|------|------|
| `空格问题` | 字母间或符号间多余的空格 |
| `下标错误` | 下标未正确闭合或格式错误 |
| `格式不统一` | 类似内容格式不一致 |
| `语法错误` | LaTeX 语法错误 |
| `语义变更` | 不小心改变了数学含义（严重） |

### 输出格式

```markdown
## 审核报告

**文件**: `{file_path}`
**审核时间**: [当前时间]

### 状态
- [ ] 通过
- [ ] 需要修正（发现 N 个问题）

### 问题清单

| 序号 | 行号 | 类型 | 内容 | 建议修复 |
|------|------|------|------|----------|
| 1 | 123 | 空格问题 | `n e t` | → `net` |
| 2 | 456 | 下标错误 | `_{i n}` | → `_{in}` |

### 通过率
80% (4/5 检查项通过)

### 建议
- 优先修复空格问题
- 审核后进行第 N 轮纠错
```

### 开始执行

请现在开始审核 `{file_path}` 的内容。
```

## 3. 纠错 Agent 模板

### 任务描述

```
## 任务：根据审核反馈修正文档

**文件路径**: `{file_path}`（使用相对路径）
**审核报告**: 详见下方

### 审核发现的问题

| 序号 | 行号 | 类型 | 内容 | 建议修复 |
|------|------|------|------|----------|
| 1 | 123 | 空格问题 | `n e t` | → `net` |
| 2 | 456 | 下标错误 | `_{i n}` | → `_{in}` |

### 执行要求

1. 逐个修复审核报告中列出的问题
2. 只修复指出的问题，不要修改其他内容
3. 修复后标记已修复的问题

### 输出格式

```markdown
## 纠错报告

**文件**: `{file_path}`

### 修复状态
| 序号 | 行号 | 状态 | 实际修复内容 |
|------|------|------|-------------|
| 1 | 123 | ✅ 已修复 | `n e t` → `\operatorname{net}` |
| 2 | 456 | ✅ 已修复 | `_{i n}` → `_{in}` |

### 无法修复的问题
（如有）
- 序号 3: [描述原因]

### 总结
共修复 N 个问题，N 个问题无法修复。
```

### 开始执行

请现在开始根据审核报告修正 `{file_path}`。
```

## 4. 协调 Agent 模板（可选）

### 任务描述

```
## 任务：协调多 Agent 文档修复流水线

**文件路径**: `{file_path}`

### 文档分析

1. 首先读取文档，确定：
   - 总行数
   - 内容结构（章节、公式密度）
   - 需要的 Agent 数量

2. 制定分段计划：

| 分段 | 行数范围 | 负责内容 |
|------|----------|----------|
| Part 1 | 1-100 | Introduction + Forward Pass |
| Part 2 | 101-200 | Backward Pass 基础 |
| ... | ... | ... |

### 执行流水线

1. **启动并行修复**
   - 同时启动 N 个修复 Agent

2. **收集修复报告**
   - 等待所有修复 Agent 完成

3. **启动审核**
   - 使用审核 Agent 检查整体结果

4. **迭代纠错**
   - 如有问题，派纠错 Agent 修正
   - 重复直到通过

5. **最终联动 pandoc-fixer**
   - 应用 pandoc-fixer 知识库
   - 验证 Pandoc 兼容性

### 输出格式

```markdown
## 流水线执行报告

**文件**: `{file_path}`
**开始时间**: [时间]
**结束时间**: [时间]

### 阶段 1: 并行修复
- 启动 Agent 数: 4
- 修改总点数: 120

### 阶段 2: 审核
- 发现问题数: 8
- 通过率: 85%

### 阶段 3: 纠错
- 纠错轮数: 2
- 最终问题数: 0

### 最终状态
✅ 文档通过所有检查
```

### 开始执行

请现在开始协调 `{file_path}` 的修复流水线。
```
