# Doc Pipeline Skill

多 Agent 协作的长文档修复与规范化流水线。

## 核心概念

### 为什么不用 Python 脚本？

Python 正则处理复杂文档时容易出错：
- 括号嵌套匹配复杂
- 跨行公式难以处理
- 修复一个问题的正则可能引入新问题

**正确做法**：使用 Agent 逐行检查 + 人工审核闭环

### 流水线架构

```
┌─────────────────────────────────────────────────────────────┐
│                    流水线式 Agent 协作                       │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐│
│  │ 修复 Agent │   │ 修复 Agent │   │ 修复 Agent │   │ 修复 Agent ││
│  │  Part 1  │   │  Part 2  │   │  Part 3  │   │  Part 4  ││
│  └────┬─────┘   └────┬─────┘   └────┬─────┘   └────┬─────┘│
│       └──────────────┴──────────────┴──────────────┘       │
│                           │                                  │
│                           ▼                                  │
│                   ┌──────────────┐                           │
│                   │   审核 Agent  │ ← 检查遗漏/错误           │
│                   └──────┬───────┘                           │
│                          │                                   │
│                          ▼                                   │
│       ┌──────────────────────────────────────────┐          │
│       │  发现问题？ → 派修复 Agent 修正 → 再审核    │          │
│       │        ↑                                   │          │
│       │        └────────── 迭代直到无问题 ─────────│          │
│       └──────────────────────────────────────────┘          │
│                           │                                  │
│                           ▼                                  │
│                    ✅ 验收通过                               │
└─────────────────────────────────────────────────────────────┘
```

## 执行流程

### ⚠️ 关键优化：预先加载 pandoc-fixer 知识库

**在启动修复 Agent 之前，必须先读取 pandoc-fixer 的 knowledge-base.md！**

这样可以：
- 避免修复 Agent 完成后再发现 pandoc-fixer 问题导致返工
- 一次修复搞定所有格式问题

```
1. 读取 pandoc-fixer/references/knowledge-base.md
2. 读取 doc-pipeline/references/fix-rules.md  
3. 合并两套规则后启动修复 Agent
4. 完成后直接验收（无需再运行 pandoc-fixer）
```

---

### 第一步：分析文档 + 备份

1. **读取文档**：确定总行数和复杂程度
2. **读取知识库**：
   - `.claude/skills/pandoc-fixer/references/knowledge-base.md`
   - `.claude/skills/doc-pipeline/references/fix-rules.md`
3. **创建备份**：
   ```python
   import shutil
   # 获取文件名和扩展名
   import os
   base, ext = os.path.splitext("原文件.md")
   shutil.copy("原文件.md", f"{base}_backup{ext}")
   ```
   - 备份文件命名：`原文件名_backup.md`
   - 存放在同一目录下
3. **判断是否需要多 Agent**：
   - < 150 行：单 Agent 直接修复
   - 150-300 行：2-4 个修复 Agent + 1 个审核 Agent
   - > 300 行：6-8 个修复 Agent + 1-2 个审核 Agent

### 第二步：并行修复

**调用修复 Agent**（多个并行）：

```
修复 Agent 指令模板（详见 references/agent-prompts.md）：
- 文件路径
- 行数范围
- 修复规则
- 输出格式要求
```

每个修复 Agent 负责一段，使用 **相对路径**（如 `notes/doc.md`）。

### 第三步：审核

**调用审核 Agent**：

```
审核 Agent 指令模板（详见 references/agent-prompts.md）：
- 检查清单
- 问题分类
- 质量评级
```

### 第四步：迭代纠错

如果审核发现问题：
1. 根据审核报告，派纠错 Agent 修复特定问题
2. 再次审核
3. 通常 2-3 轮可达标

### 第五步：联动 pandoc-fixer + 清理备份

最终使用 **pandoc-fixer** skill 进行 Pandoc 兼容性最终检查：

```
1. 读取 pandoc-fixer 的 knowledge-base.md
2. 应用知识库中的修复规则
3. 验证导出兼容性
```

**审核通过后删除备份**：
```python
import os
os.remove("原文件_backup.md")
```

## 分段策略

| 文档长度 | 建议分段数 | 备注 |
|----------|-----------|------|
| < 150 行 | 1 段 | 单 Agent 直接处理 |
| 150-300 行 | 2-4 段 | 2-4 修复 + 1 审核 |
| 300-600 行 | 4 段 | 4 修复 + 1 审核 |
| > 600 行 | 6-8 段 | 更多并行 + 多轮审核 |

**分段原则**：
- 按公式/章节自然边界分段
- 每段 80-150 行左右
- 避免在公式中间分段

## 修复规则

详见 `references/fix-rules.md`，核心规则：

| 问题类型 | 示例 | 修复方式 |
|---------|------|---------|
| 字母间空格 | `n e t` | → `net` 或 `\operatorname{net}` |
| 下标配对 | `_{i n _{j}}` | → `_{in_j}` |
| `\frac` 格式 | `\frac {x}{y}` | → `\frac{x}{y}` |
| `\text{}` 可读性 | `\text{hiddenunit}` | → `\text{hidden unit}` |
| 括号间距 | `\left( x \right)` | → `\left(x\right)` |
| 导数符号 | `^{\prime}` | → `'` 或直接用 `^{\prime}` |

## 审核清单

详见 `references/review-checklist.md`：

1. **字母间距**：无 `n e t` 等
2. **下标配对**：无 `_{i n _{j}}` 等
3. **\frac 格式**：统一 `\frac{x}{y}` 格式
4. **\text{} 可读**：单词间有空格
5. **语法正确**：括号匹配、无悬空 `\left\right`
6. **语义完整**：未误改数学含义

## 输出格式

修复 Agent 输出：

```markdown
## 修复报告

**文件**: `{file_path}`
**行数**: {start} - {end}

### 修改统计
- 修改行数: N
- 修改点数: M

### 修改详情
| 行号 | 修复前 | 修复后 |
|------|--------|--------|
| 123 | `n e t` | `net` |
| 456 | `_{i n}` | `_{in}` |

### 发现的问题
（无法修复或需要人工判断的问题）
```

审核 Agent 输出：

```markdown
## 审核报告

**文件**: `{file_path}`

### 状态
- [ ] 通过
- [x] 需要修正

### 问题清单
| 序号 | 行号 | 类型 | 内容 | 建议 |
|------|------|------|------|------|
| 1 | 123 | 空格问题 | `n e t` | → `net` |
| 2 | 456 | 下标错误 | `_{i n}` | → `_{in}` |

### 通过率
90%
```

## 关键原则

| 原则 | 说明 |
|------|------|
| **不用 Python 脚本** | 正则处理复杂文档易出错 |
| **小步快跑** | 每段修复后立即审核 |
| **审核闭环** | 修复 → 审核 → 再修复 |
| **明确边界** | 每段行数固定，减少重叠 |
| **迭代控制** | 2-3 轮通常可达标 |
| **保护语义** | 只修格式，不改数学含义 |
| **备份优先** | 处理前必须创建备份 |
| **清理善后** | 成功后删除备份文件 |

## 知识库联动

当发现新的修复模式时，**必须**追加到 pandoc-fixer 的 `references/knowledge-base.md`：

```markdown
## N. 新发现的问题类型
- **Problem**: 描述
- **Regex/Pattern**: 定位模式
- **Fix**: 修复方案
```
